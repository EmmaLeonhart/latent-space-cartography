mxbai-embed-large thinks “Hokkaidō” and “Éire” are the same word

A top-10 MTEB embedding model has a tokenizer defect that collapses nearly every word containing a diacritical mark to a single point in vector space. 147,687 confirmed cross-entity collisions on Wikidata. No standard benchmark catches it.
Last updated: 2026-04-11 · figures regenerated from live Ollama by scripts/generate_defect_page.py · collision set re-verified daily by GitHub Actions

The short version

mxbai-embed-large is a top-10 MTEB open-source embedding model. Any input containing characters outside its WordPiece vocabulary — which includes almost every diacritical mark (ō, é, ü, ł, ṣ, …) — has the offending token replaced by [UNK]. For short strings, that single [UNK] dominates the mean-pooled output, so the final 1024-dim vector is almost purely a function of “how many [UNK] tokens were in the string.”

Result: “Hokkaidō” (a Japanese island) and “Éire” (Ireland in Irish) produce identical embeddings — cosine similarity 1.000 to six decimal places. The same goes for “Zürich,” “café,” “naïve,” “Dvořák,” and most of the non-ASCII half of Wikipedia.

The paper's headline number of 147,687 collisions comes from the full Engishiki breadth-first walk (~34,000 entities, ~1,781 of them diacritical) and is the result reported in “Emergent Vector Symbolic Operations in Frozen Text Embeddings”. The daily CI reproduction above is a smaller, cheaper, self-contained experiment with a different sampling strategy — the two numbers are on different scales but they describe the same defect. What matters is the ratio: ~28% of diacritical pairs collide vs ~0.005% of ASCII-control pairs. Anywhere there are accents, mxbai-embed-large is effectively blind.

2. The Paradox: "Hokkaidō" is closer to "Éire" than to "Hokkaido"

For each word in the sample, compare two cosines: the word vs its own ASCII spelling (blue), and the word vs a different diacritical word (terracotta). A working model should put the blue bars near 1.0 (same word, same meaning) and the terracotta bars near 0.5 (unrelated words). The actual result is the exact inverse — the diacritical version of a word is closer to a random other diacritical word than it is to its own ASCII form.

3. This is a hard collapse, not gradual degradation

If the failure mode were a soft blur — diacritical marks adding noise that gradually erodes similarity — you would expect the number of colliding pairs to drop smoothly as you raise the threshold from 0.95 toward 1.0. Instead, the curve is flat all the way up to 0.99. The collisions are not approximate matches; they are points at a single location in the 1024-dim vector space.

Figure 3: 171 of 190 pairs (90%) collide at cosine ≥ 0.95 in the 20-word sample, and the count barely drops as the threshold is raised to 0.99. The collapse is binary, not gradual.

5. Why It Happens: A Four-Step Walkthrough

Step 1 — WordPiece has no character-level fallback

mxbai-embed-large inherits the bert-base-uncased WordPiece vocabulary (~30k subwords). WordPiece works by greedy left-to-right longest-match on the characters already present in its vocab. Critically, that vocab only contains basic Latin letters, digits, and a handful of punctuation marks. Characters like ō, é, or ṣ are not in the vocab at all — and WordPiece has no byte-level or character-level backoff.

Step 2 — The whole token collapses to [UNK]

When WordPiece hits a character it can’t match, it does not just skip that character — it marks the entire whitespace-delimited token as [UNK]. So “Hokkaidō” does not become ["hokkaid", "[UNK]"]; it becomes a single [UNK]. The fact that “Hokkaid” is a perfectly valid in-vocab prefix is discarded.

Step 3 — Mean pooling averages three nearly-identical vectors

mxbai-embed-large produces its sentence embedding by mean-pooling the last-layer hidden states over all non-padding tokens. For a one-word query like “Hokkaidō”, the model only sees three tokens: 

"Hokkaidō"  →  [CLS]  [UNK]  [SEP]
"Éire"      →  [CLS]  [UNK]  [SEP]
"Zürich"    →  [CLS]  [UNK]  [SEP]
"café"      →  [CLS]  [UNK]  [SEP]
"Dvořák"    →  [CLS]  [UNK]  [SEP]

Every one of those strings produces exactly the same token sequence. There is literally no input signal left for the transformer to condition on: the final mean-pooled vector is a deterministic function of (CLS, UNK, SEP) and nothing else.

Step 4 — The [UNK] vector dominates

Because mxbai was trained almost exclusively on English web text, the [UNK] token embedding sits at a peculiar point in representation space: it has moderate cosine similarity (~0.3–0.5) to most English content, but perfect (1.000) cosine similarity to any other string that also pooled into (CLS, UNK, SEP). This is not gradual degradation with diacritic density — it is a hard collapse to a single point.

Why no benchmark caught it

MTEB’s retrieval, clustering, and STS tasks are dominated by English ASCII text. The failure mode only fires when the query or document contains a non-ASCII character, which is rare enough in MTEB to disappear into the aggregate score. On the MTEB leaderboard, mxbai-embed-large ranks in the top 10. On any Wikidata entity label set, it has a ~15% catastrophic collision rate.

7. Reproducing This

There are two reproduction scripts in the repo, for different purposes:

scripts/wikidata_collision_scan.py — the full statistical experiment. Queries Wikidata for ~700 diacritical settlements and ~300 ASCII controls, embeds everything, computes the pairwise collision rate. This is what runs on the daily GitHub Actions cron. Requires numpy plus Ollama.

scripts/demo_collisions.py — a tiny, dependency-free proof of the collision phenomenon on 25 hand-picked test pairs. Stdlib only. Good for a 10-second smoke test. 

# 1. Install Ollama and pull the model
ollama pull mxbai-embed-large

# 2. Clone the repo
git clone https://github.com/EmmaLeonhart/latent-space-cartography.git
cd latent-space-cartography

# 3a. Tiny smoke test (no deps)
python scripts/demo_collisions.py

# 3b. Full Wikidata reproduction (~5 min, needs numpy)
pip install numpy
python scripts/wikidata_collision_scan.py

The full scan typically finds on the order of 60,000 cross-entity collisions in a sample of ~700 diacritical entities, while the ASCII control group produces essentially none. The exact counts drift slightly run-to-run because Ollama's embedding pipeline has ~1e−4 float-point non-determinism and the SPARQL endpoint returns different sub-samples for the same query, but the collision rate is stable to within a percentage point. Summary JSON from each CI run is uploaded as a GitHub Actions artifact.

8. What Should Be Done

  1. Tokenizer fix: Replace unknown-character fallback with character-level tokenization or byte-level BPE
  2. Benchmark gap: MTEB should include diacritical/multilingual string pairs as a robustness check
  3. User workaround: Strip diacritical marks before embedding (loses linguistic information but prevents collisions)
  4. Model choice: Use models with byte-level tokenizers (e.g., nomic-embed-text) for non-ASCII data

9. Context

This defect was discovered during the Latent Space Cartography project, which applies Vector Symbolic Architecture (VSA) analysis to frozen text embeddings. When probing Wikidata entity embeddings for relational structure, the collision pattern was unmistakable: 147,687 cross-entity pairs at cosine ≥ 0.95, all involving diacritical text.

The full analysis is documented in our paper: "Emergent Vector Symbolic Operations in Frozen Text Embeddings" (clawRxiv 2604.00859).
